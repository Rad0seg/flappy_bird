# Отчёт по семестровой работе  
## «Flappy Bird на чистом C + Win32 API»

## 1. Общая информация

Проект представляет собой 2D-игру **Flappy Bird**, реализованную на:

- **Язык:** C99
- **API:** чистый Win32 API
- **Графика:** GDI
- **Без использования:** C++, STL, SDL, SFML, OpenGL, DirectX wrappers и других сторонних игровых библиотек

Игра содержит:

- Классическую механику Flappy Bird
- Прогрессивное увеличение сложности
- Систему баффов и дебаффов
- Сохранение рекорда
- Смену времени суток
- Пиксельную графику
- Двойную буферизацию без мерцания
- Паузу с обратным отсчётом перед продолжением игры

---

## 2. Архитектура проекта

Проект разделён на независимые модули.

### 2.1 Модуль logic / core

Файлы:

- `game_logic.h`
- `game_logic.c`

Отвечает за:

- Игровое состояние `GameState`
- Физику птицы
- Гравитацию
- Прыжок игрока
- Движение труб
- Коллизии
- Подсчёт очков
- Прогрессивную сложность
- Баффы и дебаффы
- Таймеры эффектов
- Смену времени суток
- Сохранение и загрузку рекорда

Важно:

В модуле логики нет:

- `HWND`
- `HDC`
- `HBITMAP`
- GDI-объектов
- прямой зависимости от Win32-отрисовки

Логика работает только со структурами состояния и обычными C-функциями.

---

### 2.2 Модуль ui / interface

Файлы:

- `ui.h`
- `ui.c`

Отвечает за:

- Создание окна Win32
- Регистрацию оконного класса
- Обработку сообщений Windows
- Обработку клавиатуры
- Отрисовку через GDI
- Двойную буферизацию
- Отображение HUD
- Отрисовку меню
- Отрисовку паузы
- Отрисовку экрана Game Over
- Отрисовку птицы, труб, баффов, земли, облаков и звёзд

Все GDI-ресурсы освобождаются в `ui_cleanup()` через:

- `DeleteObject`
- `DeleteDC`
- `ReleaseDC`
- `DestroyWindow`

Утечек GDI-объектов нет.

---

### 2.3 Точка входа

Файл:

- `main.c`

Отвечает за:

- Создание структуры `GameState`
- Создание структуры `UIContext`
- Инициализацию логики
- Инициализацию интерфейса
- Запуск игрового цикла
- Очистку ресурсов перед выходом

Основной код программы не находится в `main.c`, чтобы сохранить модульность.

## 3. Игровой цикл

Игровой цикл находится в модуле `ui.c`.

Для расчёта времени используется:

- `QueryPerformanceFrequency`
- `QueryPerformanceCounter`

`SetTimer` для главного игрового цикла не используется.

Цикл работает по следующей схеме:

```c
while (running) {
    while (PeekMessageA(&msg, NULL, 0, 0, PM_REMOVE)) {
        TranslateMessage(&msg);
        DispatchMessageA(&msg);
    }

    QueryPerformanceCounter(&now);
    delta_time = ...;

    game_update(state, delta_time);
    ui_render(ctx, state);
}
```

Особенности игрового цикла:

- Сообщения Windows обрабатываются через `PeekMessageA`
- Обработка сообщений не блокирует игру
- Время кадра считается через высокоточный таймер
- `delta_time` ограничивается сверху, чтобы избежать скачков физики
- Логика и отрисовка разделены

Ограничение `delta_time` нужно для ситуаций, когда окно временно не обновлялось, например при перетаскивании окна пользователем.

## 4. Структура состояния игры

Вся игра хранится в структуре:

```c
typedef struct GameState
```

Она содержит:

- Птицу игрока
- Массив труб
- Массив баффов и дебаффов
- Активный эффект
- Текущий счёт
- Лучший счёт
- Текущее состояние игры
- Таймеры
- Скорость труб
- Расстояние между трубами
- Текущую гравитацию
- Текущее время суток
- Флаг исчезновения труб

`GameState` передаётся по указателю между модулями.

Это позволяет:

- Не использовать глобальное состояние в логике
- Хранить все данные игры в одном месте
- Упростить перезапуск игры
- Упростить отладку

## 5. Система состояний игры

В игре используется конечный автомат состояний:

```c
typedef enum {
    GAME_STATE_MENU,
    GAME_STATE_PLAYING,
    GAME_STATE_PAUSED,
    GAME_STATE_COUNTDOWN,
    GAME_STATE_GAME_OVER
} GameStateType;
```

Состояния:

- `GAME_STATE_MENU` — главное меню
- `GAME_STATE_PLAYING` — активная игра
- `GAME_STATE_PAUSED` — пауза
- `GAME_STATE_COUNTDOWN` — обратный отсчёт 3-2-1
- `GAME_STATE_GAME_OVER` — конец игры

Такой подход исключает конфликтующие состояния, например ситуацию, когда игра одновременно находится и в паузе, и в Game Over.

---

## 6. Система прогрессивной сложности

Сложность увеличивается каждые 10 пройденных труб.

Усложнение происходит за счёт:

- Увеличения скорости труб
- Очень медленного увеличения горизонтального расстояния между трубами

Промежуток между верхней и нижней трубой не увеличивается.

Он остаётся постоянным:

```c
PIPE_GAP_BASE = 180
```

Формула сложности:

```c
level = pipes_passed / 10;
speed = 3.0f + level * 0.5f;
distance = 250.0f + level * 3.0f;
```

Пример прогрессии:

| Пройдено труб | Скорость | Расстояние между трубами  |
|---------------|----------|---------------------------|
| 0–9           | 2.5      | 250 px                    |
| 10–19         | 3.5      | 253 px                    |
| 20–29         | 4.0      | 256 px                    |
| 30–39         | 4.5      | 259 px                    |
| 40–49         | 5.0      | 262 px                    |
| 50+           | 5.5+     | 265+ px                   |

С 50 очков игра становится заметно сложной.

## 7. Система баффов и дебаффов

В игре реализованы 3 эффекта.

| Символ | Название    | Тип    | Эффект                                |
|--------|-------------|--------|---------------------------------------|
| S      | Mini Bird   | Бафф   | Птица уменьшается на 30%              |
| C      | Jump Points | Бафф   | Трубы исчезают, очки даются за прыжки |
| H      | Heavy       | Дебафф | Гравитация увеличивается в 1.5 раза   |

### 7.1 Mini Bird

При подборе эффекта:

```c
bird.size_multiplier = 0.7f;
```

Птица становится меньше, поэтому игроку проще пролетать между трубами.

Эффект длится 5 секунд.

---

### 7.2 Jump Points

При подборе эффекта:

- Все трубы исчезают
- Коллизии с трубами временно отключаются
- За каждое нажатие `SPACE` игрок получает +1 очко
- После завершения эффекта трубы появляются снова

После окончания эффекта трубы пересоздаются функцией:

```c
respawn_pipes(...)
```

Трубы появляются ближе к экрану, чтобы игрок не ждал слишком долго продолжения игры.

---

### 7.3 Heavy

При подборе эффекта:

```c
current_gravity = GRAVITY * 1.5f;
```

Птица начинает быстрее падать.

Эффект длится 5 секунд.

---

## 8. Коллизии

Для столкновения птицы с трубами используется AABB-коллизия.

Проверяются:

- Левая граница птицы
- Правая граница птицы
- Верхняя граница птицы
- Нижняя граница птицы
- Границы труб

Пример логики:

```c
if (bird_right > pipe_left && bird_left < pipe_right) {
    if (bird_top < pipe_top_height) {
        return 1;
    }

    if (bird_bottom > bottom_pipe_top) {
        return 1;
    }
}
```

Для подбора баффов используется круговая коллизия:

```c
dist < bird_radius + powerup_radius
```

Такой способ удобнее для объектов небольшого размера.

---

## 9. Двойная буферизация

Для устранения мерцания используется двойная буферизация.

Используются функции Win32 GDI:

- `GetDC`
- `CreateCompatibleDC`
- `CreateCompatibleBitmap`
- `SelectObject`
- `BitBlt`

Схема работы:

1. Создаётся буфер в памяти
2. Весь кадр рисуется в этот буфер
3. Готовый кадр одной операцией копируется на экран через `BitBlt`

Это позволяет избежать мерцания при перерисовке фона, труб, птицы и интерфейса.

---

## 10. Графика

Графика реализована программно через GDI.

Используются:

- `Rectangle`
- `Ellipse`
- `FillRect`
- `TextOutA`
- `CreateSolidBrush`
- `CreatePen`
- `CreateFontA`

Графические элементы:

- Пиксельная птица
- Трубы с простым 3D-эффектом
- Плавно движущиеся облака
- Ночное небо со звёздами
- Земля с кустами
- Баффы с пульсацией
- HUD с текущим счётом и активным эффектом

BMP-спрайты не используются.

## 11. Сохранение рекорда

Лучший результат сохраняется в файл:

```text
flappy_best.dat
```

Для работы с файлом используются стандартные функции C:

- `fopen`
- `fread`
- `fwrite`
- `fclose`

После завершения чтения или записи файл всегда закрывается.

Если файл отсутствует или содержит некорректные данные, рекорд считается равным 0.

---

## 12. Управление памятью

Динамическая память через `malloc` не используется.

Все массивы имеют фиксированный размер:

- `pipes[MAX_PIPES]`
- `powerups[MAX_POWERUPS]`

Все GDI-ресурсы освобождаются.

Примеры парных вызовов:

| Создание | Освобождение |
|---------|--------------|
| `CreateSolidBrush` | `DeleteObject` |
| `CreatePen` | `DeleteObject` |
| `CreateFontA` | `DeleteObject` |
| `CreateCompatibleDC` | `DeleteDC` |
| `GetDC` | `ReleaseDC` |
| `CreateCompatibleBitmap` | `DeleteObject` |

Функция очистки:

```c
ui_cleanup(...)
```

вызывается перед завершением программы.

Утечек памяти и GDI-ресурсов нет.

## 13. Трудности при разработке

### 13.1 Мерцание изображения

Проблема:

При прямой отрисовке в `HDC` окна изображение мерцало.

Решение:

Была реализована двойная буферизация через `CreateCompatibleDC`, `CreateCompatibleBitmap` и `BitBlt`.

---

### 13.2 Передача GameState в WndProc

Проблема:

Оконная процедура Win32 имеет фиксированную сигнатуру, поэтому в неё нельзя напрямую передать пользовательские данные.

Решение:

В модуле `ui.c` используется минимально необходимый `static GameState*`.

Он нужен только для обработки клавиатуры в `WndProc`.

Игровая логика при этом не содержит глобальных переменных и не зависит от Win32.

---

### 13.3 Баланс сложности

Проблема:

Если сильно увеличивать расстояние между трубами, игра становилась легче со временем.

Решение:

Горизонтальное расстояние между трубами увеличивается очень медленно.

Основной рост сложности идёт через скорость труб.

---

### 13.4 Бафф Jump Points

Проблема:

Изначально во время исчезновения труб очки начислялись автоматически, и игрок просто ждал окончания эффекта.

Решение:

Механика была изменена.

Теперь во время эффекта игрок получает очки только за нажатия `SPACE`.

Это делает бафф активным и более интересным.

---

### 13.5 Появление труб после исчезновения

Проблема:

После окончания эффекта исчезновения труб новые трубы появлялись слишком поздно.

Решение:

Трубы пересоздаются ближе к экрану:

```c
WINDOW_WIDTH + 100.0f
```

Поэтому игра быстро возвращается в обычный режим.
